# Реконструкция User Journey: Учебное заведение (Institution)

Данный документ представляет собой точную, технически достоверную карту поведения пользователя с ролью `institution` внутри существующей платформы. Описание базируется исключительно на реализованной архитектуре, компонентах (React/Vite), схеме БД (Supabase PostgreSQL) и настроенных политиках RLS.

---

## 1. ПОЛНЫЙ USER JOURNEY (ОСНОВА)

### Этап 1: Регистрация и Верификация
* **Роли:** `institution` (заявитель), `admin` (проверяющий).
* **Точка входа:** Страница регистрации / Auth UI.
* **Условия начала:** Пользователь не авторизован.
* **Действия:** Ввод email/пароля, создание профиля.
* **Сущности:** Создается запись в `auth.users` (Supabase) и триггером создается запись в `public.users`.
* **Данные:** `email`, `role = 'institution'`, `status = 'Pending'`.
* **Проверки:** Уникальность email, валидация пароля.
* **Статусы:** `Pending` (ожидание проверки администратором).
* **Ошибки:** Email уже существует.
* **Завершение этапа:** Администратор в `AdminDashboard` меняет статус пользователя на `Active`.

### Этап 2: Наполнение каталога (Управление программами)
* **Роли:** `institution`, `admin`.
* **Точка входа:** `ProgramsList.tsx`.
* **Условия начала:** `auth.uid()` активен, `role == 'institution'`.
* **Действия:** Нажатие "Add Program", заполнение формы, сохранение.
* **Сущности:** Создается запись в `public.programs`.
* **Данные:** `name`, `level`, `duration`, `tuition_fee`, `application_fee`, `requirements`, `intakes`.
* **Автоматические процессы:** При `INSERT` программа получает `status = 'Pending'`.
* **Ошибки:** Нарушение RLS (попытка создать программу от чужого `institution_id`).
* **Завершение этапа:** Администратор одобряет программу (`status = 'Active'`), после чего она становится видна рекрутерам.

### Этап 3: Обработка заявок (Core Pipeline)
* **Роли:** `institution`, `recruiter` (косвенно).
* **Точка входа:** `ApplicationsList.tsx` (режимы Table / Kanban).
* **Условия начала:** Рекрутер создал `application` с `institution_id`, равным ID данного вуза.
* **Действия:** Просмотр списка, открытие `ApplicationCard.tsx`, изменение статуса.
* **Сущности:** Чтение/Обновление `public.applications`.
* **Проверки:** RLS проверяет `institution_id == auth.uid()`.
* **Статусы заявки:** `Submitted` -> `Reviewing` -> `Action Required` -> `Approved` / `Rejected`.
* **Автоматические процессы (Триггеры):** При изменении `applications.status` срабатывает PostgreSQL триггер `sync_student_status`, который вызывает функцию `update_student_status_from_app()` и синхронизирует статус в таблице `students` (например, `Approved` -> `Payment`).
* **Завершение этапа:** Заявка переведена в терминальный статус (`Approved` или `Rejected`).

### Этап 4: Коммуникация по заявке (Чат)
* **Точка входа:** `ApplicationCard.tsx` -> Кнопка "Chat with Partner" / "Action Required".
* **Условия начала:** Необходимость запросить документы или ответить рекрутеру.
* **Действия:** Ввод текста, отправка.
* **Сущности:** `public.chats` (создается лениво, если нет), `public.messages`.
* **Системные реакции:** Если чата нет, выполняется `INSERT INTO chats (application_id, institution_id, recruiter_id)`. Затем `INSERT INTO messages`. Обновляется `chats.updated_at`.

### Этап 5: Финансы и Зачисление
* **Точка входа:** `ApplicationCard.tsx`.
* **Условия начала:** Заявка в статусе `Approved` (или легаси `Accepted`).
* **Действия:** Нажатие кнопок "Confirm Payment", "Issue Documents".
* **Сущности:** Обновление `applications.status`.
* **Статусы:** `Paid`, `Documents Issued`.
* **Автоматические процессы:** Переход в `Paid` логически разблокирует начисление `Pending Balance` для рекрутера (согласно финансовой логике платформы).

## 2. МИКРО-ШАГИ (ГРАНУЛЯРНОСТЬ МАКСИМУМ)

**Сценарий: Обработка новой заявки с запросом документов**

1. Пользователь открывает `/dashboard/applications`.
2. Система делает `SELECT * FROM applications WHERE institution_id = auth.uid()`.
3. Пользователь видит заявку в колонке `Submitted` (Kanban) или в таблице.
4. Пользователь кликает на карточку заявки.
5. Система рендерит `ApplicationCard.tsx`, передавая объект `application`.
6. Пользователь нажимает кнопку "Action Required" в блоке Status Management.
7. Система открывает модальное окно `isActionModalOpen = true`.
8. Пользователь вводит текст в `textarea` (например, "Нужен скан паспорта").
9. Пользователь нажимает "Send & Update Status".
10. Система блокирует UI (`isUpdating = true`).
11. Система выполняет `UPDATE applications SET status = 'Action Required' WHERE id = application.db_id`.
12. PostgreSQL триггер `sync_student_status` перехватывает UPDATE, находит связанного `student_id` и делает `UPDATE students SET status = 'Preparation'`.
13. Система проверяет наличие чата: `SELECT id FROM chats WHERE application_id = ... LIMIT 1`.
14. Чата нет. Система выполняет `INSERT INTO chats (application_id, institution_id, recruiter_id) RETURNING id`.
15. Система выполняет `INSERT INTO messages (chat_id, sender_id, text) VALUES (...)`.
16. Система закрывает модалку, обновляет локальный стейт, вызывает `onStatusUpdate()`.
17. Карточка в Kanban-доске перемещается в колонку `Action Required`.

## 3. СУЩНОСТИ И ДАННЫЕ

*   **`users`**:
    *   *Поля:* `id` (UUID, PK), `email`, `role` ('institution'), `status` ('Pending', 'Active', 'Rejected'), `name`, `country`, `city`.
    *   *Жизненный цикл:* Создается при регистрации -> `Pending` -> Админ делает `Active`.
*   **`programs`**:
    *   *Поля:* `id`, `institution_id` (FK -> users.id), `name`, `level`, `duration`, `tuition_fee`, `status`.
    *   *Жизненный цикл:* Создается вузом -> `Pending` -> Админ делает `Active`.
*   **`applications`**:
    *   *Поля:* `id`, `student_id`, `program_id`, `recruiter_id`, `institution_id`, `status`.
    *   *Жизненный цикл:* `Submitted` -> `Reviewing` -> `Action Required` -> `Approved` -> `Paid` -> `Documents Issued`. (Или `Rejected`).
*   **`chats` & `messages`**:
    *   Связующие сущности для коммуникации. Чат жестко привязан к `application_id`.

*Факт системы:* В текущей реализации 1 аккаунт = 1 учебное заведение. Сложной иерархии (Admin -> Officer) внутри самого вуза в БД пока нет, используется единый аккаунт с `role = 'institution'`.

---

## 4. РОЛИ И ДОСТУПЫ

*   **Роль `institution`**:
    *   Назначается при регистрации (выбор типа аккаунта).
    *   **Доступы (RLS):**
        *   `programs`: `SELECT`, `INSERT`, `UPDATE` где `institution_id = auth.uid()`.
        *   `applications`: `SELECT`, `UPDATE` где `institution_id = auth.uid()`. Не может делать `INSERT` (это делает `recruiter`) или `DELETE`.
        *   `students`: Может делать `SELECT` студентов, привязанных к их заявкам (через join).
        *   `chats`/`messages`: Доступ к записям, где `institution_id = auth.uid()`.
*   **Ограничения:** Вуз не видит заявки других вузов, не видит финансовый баланс рекрутеров (`payouts`).

---

## 5. СИСТЕМНАЯ ЛОГИКА

*   **Бизнес-правила:**
    *   Вуз не может получать заявки, пока его статус `Pending`.
    *   Программы вуза не видны рекрутерам, пока статус программы `Pending`.
*   **Триггеры (Events -> Actions):**
    *   `sync_student_status`: Автоматическая синхронизация. Если вуз ставит заявке `Approved`, студент получает статус `Payment`. Если `Documents Issued` -> студент `Enrolled`.
*   **Ленивая инициализация чатов:** Чат не создается в момент подачи заявки. Он создается в `ApplicationCard.tsx` только при попытке отправить первое сообщение (функции `handleChatAction` и `submitActionRequired`).

## 6. ВСЕ СЦЕНАРИИ

### 6.1. Основной (Happy Path)
Вход -> `ApplicationsList` -> Перенос карточки в `Reviewing` -> Проверка -> Нажатие `Approve` -> Ожидание оплаты -> Нажатие `Confirm Payment` -> Нажатие `Issue Documents`.

### 6.2. Альтернативные ветки
*   **Запрос документов:** Вуз переводит в `Action Required` -> Пишет сообщение -> Ждет. Рекрутер читает сообщение в своем интерфейсе, догружает данные, пишет ответ. Вуз видит ответ, переводит в `Reviewing`, затем `Approve`.
*   **Отказ:** Вуз нажимает `Reject Application`. Заявка блокируется для дальнейших изменений. Статус студента меняется на `Rejected` (через триггер).

### 6.3. Ошибки и сбои
*   **Race Condition в чатах:** Если два пользователя (или вуз и рекрутер одновременно) попытаются написать первое сообщение, может создаться два чата. *Как решено в коде:* В `ApplicationCard.tsx` перед `INSERT INTO chats` делается `SELECT id ... maybeSingle()`.
*   **Сбой RLS:** Если вуз попытается обновить чужую заявку (например, через подмену ID в API), Supabase RLS вернет ошибку, UI покажет `alert('Error updating application status')`.

### 6.4. Повторное использование
При повторном входе система загружает `Dashboard.tsx`, где отображается статистика: количество `Pending Reviews`, конверсия. Пользователь использует фильтры и переключатель видов (Table/Kanban) в `ApplicationsList.tsx`. Состояние переключателя локальное.

## 7. ONBOARDING

В текущей архитектуре системы **выделенного пошагового Onboarding Wizard (с progress bar) нет**.
*   **Фактический флоу:** Пользователь регистрируется -> Видит пустой дашборд и статус `Pending` -> Ждет аппрува админа -> Идет в `Programs` -> Нажимает `Add Program`.
*   **Условие завершения:** Добавление первой программы и ее аппрув админом.

## 8. ТОЧКИ СЛОЖНОСТИ

1.  **Двойной аппрув (Узкое горлышко):** Вуз зависит от `admin` дважды. Сначала для активации аккаунта, затем для активации каждой созданной программы. Если админ не в сети, вуз "застревает".
2.  **Синхронизация статусов:** Триггер `update_student_status_from_app` жестко завязан на текстовые значения статусов (`Approved`, `Action Required`). Если в UI (`ApplicationCard.tsx`) изменить текст статуса, триггер в PostgreSQL перестанет корректно мапить статусы студента.
3.  **Отсутствие файлов в БД:** В текущей схеме `applications` нет жестко типизированных полей для файлов (Offer Letter, Invoice). Логика подразумевает их наличие, но на уровне БД это пока не формализовано (в UI есть заглушка "Document management is currently being updated").

---

## 9. КРИТИЧЕСКИЕ ТОЧКИ СИСТЕМЫ

1.  **`users.status = 'Active'`**: Без этого вуз парализован.
2.  **`programs.status = 'Active'`**: Без этого рекрутеры не могут подать заявки.
3.  **`applications.status`**: Это главный стейт-машин системы. От него зависят чаты, статусы студентов и будущие финансовые выплаты (`Pending Balance`).
4.  **Точка активации:** Перевод первой заявки из `Submitted` в `Reviewing`.

---

## 10. ФИНАЛ

1.  **Краткая карта:** Регистрация -> Ожидание Админа -> Создание Программ -> Ожидание Админа -> Получение Заявок -> Kanban (Review -> Action -> Approve -> Paid) -> Зачисление.
2.  **Самые сложные участки:** Логика создания чатов "на лету" и синхронизация статусов заявок со статусами студентов через SQL-триггеры.
3.  **Самые важные сущности:** `applications` (ядро платформы), `programs` (основа для заявок).
4.  **Ключевые зависимости:** `applications` зависит от `programs` и `users` (recruiter & institution). Изменение `applications` триггерит изменения в `students`.
5.  **Потенциально слабые места реализации:**
    *   Отсутствие многопользовательского доступа внутри одного вуза (1 аккаунт = 1 вуз).
    *   Жесткая привязка логики к текстовым строкам статусов в БД (вместо ENUM или ID статусов).
    *   Ручное управление финансовыми статусами (`Paid`) без интеграции с платежным шлюзом, что может привести к человеческим ошибкам.
