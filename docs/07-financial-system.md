# Спецификация: Финансовый модуль (Commissions & Payouts)

## 1. Обзор модуля
Финансовый модуль отвечает за расчет, начисление и выплату комиссионных вознаграждений рекрутерам. Платформа выступает гарантом сделки: школа подтверждает оплату от студента, платформа фиксирует комиссию на балансе рекрутера, рекрутер запрашивает вывод средств.

## 2. Жизненный цикл комиссии

### Этап 1: Установка комиссии (Program Creation)
- Когда школа создает программу, она указывает `tuition_fee` (стоимость обучения).
- Школа **не может** сама установить процент комиссии.
- Программа уходит на модерацию к Администратору.
- Администратор при одобрении программы (кнопка `Approve`) устанавливает `commission` (процент, например, 15%).

### Этап 2: Заморозка (Pending Balance)
- Рекрутер подает заявку.
- Школа переводит заявку в статус `Accepted` (студенту выдан оффер).
- В этот момент потенциальная комиссия отображается у рекрутера как `Pending Balance`. Это мотивирует рекрутера "дожать" студента до оплаты.
- *Формула:* `Pending Balance = SUM(tuition_fee * (commission / 100))` для всех заявок в статусе `Accepted`.

### Этап 3: Начисление (Available Balance)
- Студент оплачивает обучение напрямую в школу.
- Школа переводит статус заявки в `Paid`.
- Система автоматически создает запись в таблице `transactions`:
  - `recruiter_id`: ID рекрутера.
  - `application_id`: ID заявки.
  - `amount`: Рассчитанная сумма комиссии.
  - `type`: `'Commission'`.
  - `status`: `'Available'`.
- Сумма переходит из `Pending Balance` в `Available Balance` рекрутера.

### Этап 4: Запрос на вывод (Payout Request)
- Рекрутер нажимает `Request Payout` в своем кабинете.
- Указывает сумму (не больше Available Balance) и банковские реквизиты.
- Создается запись в `payout_requests` со статусом `Pending`.
- Создается запись в `transactions` с типом `'Withdrawal'` и статусом `'Pending'` (сумма отрицательная).
- `Available Balance` рекрутера уменьшается на запрошенную сумму.

### Этап 5: Выплата (Processing by Admin)
- Администратор видит запрос в панели `Payout Requests`.
- Админ осуществляет реальный банковский перевод по указанным реквизитам (вне платформы).
- После успешного перевода админ нажимает `Mark as Processed` в системе.
- Статус в `payout_requests` меняется на `Processed`.
- Статус соответствующей транзакции вывода меняется на `Completed`.

## 3. Структура БД для финансов

### Таблица `transactions`
| Поле | Тип | Описание |
| :--- | :--- | :--- |
| `id` | `uuid` | Primary Key |
| `recruiter_id` | `uuid` | Владелец баланса |
| `application_id`| `uuid` | Ссылка на заявку (только для type='Commission') |
| `amount` | `numeric` | Сумма (положительная для начислений, отрицательная для выводов) |
| `type` | `text` | `'Commission'` или `'Withdrawal'` |
| `status` | `text` | `'Pending'`, `'Available'`, `'Completed'` |
| `created_at` | `timestamptz` | Дата транзакции |

### Таблица `payout_requests`
| Поле | Тип | Описание |
| :--- | :--- | :--- |
| `id` | `uuid` | Primary Key |
| `recruiter_id` | `uuid` | Кто запросил |
| `amount` | `numeric` | Сумма к выводу |
| `bank_details` | `text` | Реквизиты (IBAN, SWIFT, Beneficiary Name) |
| `status` | `text` | `'Pending'`, `'Processed'`, `'Rejected'` |
| `created_at` | `timestamptz` | Дата запроса |

## 4. Безопасность и RLS
- **Transactions:** Рекрутер может делать только `SELECT` своих транзакций (`auth.uid() = recruiter_id`). Никаких `INSERT` или `UPDATE` со стороны рекрутера.
- **Payout Requests:** Рекрутер может делать `INSERT` (создать запрос) и `SELECT` своих запросов. `UPDATE` (изменение статуса) доступно **только администратору**.
- **Валюта:** В текущей версии платформы базовой валютой для расчетов комиссий считается валюта программы (например, EUR или USD). В будущем планируется введение единой валюты расчетов с конвертацией.
