# Спецификация: Панель Администратора (Admin Dashboard)

## 1. Обзор модуля
Панель администратора — это центр управления всей платформой. Администратор (роль `admin`) отвечает за модерацию контента, верификацию пользователей, установку финансовых условий (комиссий) и разрешение споров.

## 2. Основные разделы и функции

### 2.1. Верификация пользователей (User Verification)
Все новые рекрутеры при регистрации получают статус `Pending`. Они не могут подавать заявки до одобрения администратором.
- **Интерфейс:** Таблица со списком пользователей в статусе `Pending`.
- **Данные для проверки:** Имя, Email, Название агентства. В будущем — загруженные KYC документы (сертификаты агентства).
- **Действия:** 
  - `Approve`: Статус меняется на `Active`. Рекрутер получает доступ к каталогу и подаче заявок.
  - `Reject`: Статус меняется на `Rejected`. Доступ блокируется.

### 2.2. Модерация программ (Program Approval)
Когда учебное заведение создает новую программу, она получает статус `Pending`.
- **Интерфейс:** Таблица программ, ожидающих модерации.
- **Действия:**
  - Просмотр всех деталей программы (требования, стоимость, страны).
  - **Установка комиссии:** При нажатии `Approve` администратор **обязан** указать процент комиссии (Commission %), который будет получать рекрутер за эту программу. Без этого программа не может стать `Active`.
  - `Reject`: Программа отклоняется с указанием причины (опционально).

### 2.3. Управление заявками (Applications Overview)
Администратор имеет доступ "только для чтения" (Read-Only) ко всем заявкам на платформе (Global View).
- **Интерфейс:** Kanban-доска или таблица со всеми заявками.
- **Фильтры:** По школе, по рекрутеру, по статусу.
- **Цель:** Мониторинг конверсии и выявление "зависших" заявок (например, школа долго не отвечает).

### 2.4. Финансовый контроль (Payouts & Transactions)
Управление запросами на вывод средств от рекрутеров.
- **Интерфейс:** Вкладка "Payout Requests".
- **Процесс:**
  1. Рекрутер запрашивает вывод доступных средств.
  2. Запрос появляется у админа со статусом `Pending`.
  3. Админ проводит транзакцию через внешний банк/платежную систему по указанным реквизитам.
  4. Админ нажимает `Mark as Processed` в системе.
  5. Баланс рекрутера обновляется, транзакция переходит в статус `Completed`.

### 2.5. Разрешение споров (Dispute Resolution / Inbox)
Администратор имеет доступ к системе сообщений (Inbox) с особыми правами.
- **Мониторинг:** Админ видит список всех активных чатов между рекрутерами и школами.
- **Вмешательство:** Админ может нажать кнопку `Join Chat` (добавляет `admin_id` в таблицу `chats`).
- **Системные уведомления:** При входе админа в чат, система автоматически отправляет сообщение: `__SYSTEM_NOTIFICATION__ Administrator has joined the chat`.
- **Действия в чате:** Админ может писать сообщения для урегулирования конфликтов (например, если школа отказывается выплачивать комиссию за зачисленного студента).
- **Выход:** Кнопка `Leave Chat` убирает админа из участников и отправляет соответствующее системное уведомление.

## 3. Технические особенности
- **RLS Bypass:** В Supabase политики RLS для роли `admin` настроены так, чтобы разрешать `SELECT`, `UPDATE` и `DELETE` для всех строк во всех таблицах.
- **UI Компоненты:** Использование `motion/react` для анимации появления новых запросов на верификацию. Цветовая схема админки базируется на цвете `indigo-600`.
