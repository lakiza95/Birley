# Спецификация: Система сообщений (Inbox & Chat)

## 1. Обзор архитектуры
Система сообщений на платформе построена на базе **Supabase Realtime (Channels)**. Она обеспечивает мгновенный обмен сообщениями между Рекрутерами (Recruiters), Учебными заведениями (Institutions) и Администраторами (Admins).

Главная архитектурная особенность: **Каждый чат строго привязан к конкретной заявке (Application).** Платформа не поддерживает абстрактные чаты "пользователь-пользователь". Общение всегда контекстно.

## 2. Структура данных

### 2.1. Таблица `chats`
- `id` (uuid)
- `application_id` (uuid) - Ссылка на заявку.
- `recruiter_id` (uuid) - Ссылка на профиль рекрутера.
- `institution_id` (uuid) - Ссылка на профиль школы.
- `admin_id` (uuid, nullable) - Ссылка на профиль админа, если он подключился к спору.
- `updated_at` (timestamptz) - Обновляется при каждом новом сообщении для сортировки списка чатов.

### 2.2. Таблица `messages`
- `id` (uuid)
- `chat_id` (uuid)
- `sender_id` (uuid)
- `text` (text)
- `created_at` (timestamptz)

## 3. Логика работы (Real-time)

### 3.1. Подписка на список чатов
Компонент `Inbox.tsx` при монтировании подписывается на изменения в таблице `chats`:
```typescript
const chatsChannel = supabase
  .channel('public:chats')
  .on('postgres_changes', { event: '*', schema: 'public', table: 'chats' }, fetchChats)
  .subscribe();
```
Это позволяет мгновенно поднимать чат наверх списка, когда в него приходит новое сообщение (за счет триггера, обновляющего `updated_at`).

### 3.2. Подписка на сообщения
При выборе конкретного чата создается подписка на таблицу `messages` с фильтрацией по `chat_id`:
```typescript
const channel = supabase
  .channel(`chat:${selectedChatId}`)
  .on('postgres_changes', { 
    event: 'INSERT', 
    schema: 'public', 
    table: 'messages',
    filter: `chat_id=eq.${selectedChatId}`
  }, (payload) => {
    // Добавление нового сообщения в стейт
  })
  .subscribe();
```

## 4. Системные уведомления (System Notifications)
Для логирования важных событий в чате используется механизм системных сообщений.
- **Формат:** Текст сообщения начинается с префикса `__SYSTEM_NOTIFICATION__`.
- **Рендеринг:** На фронтенде такие сообщения парсятся и отображаются не как обычные пузыри чата (bubbles), а как центрированные серые плашки (например, "Administrator has joined the chat").
- **Сценарии использования:**
  - Админ нажал `Add Admin` / `Join Chat`.
  - Админ нажал `Leave Chat`.
  - Изменился статус заявки (опционально, например: "Application status changed to Accepted").

## 5. Разрешение споров (Dispute Resolution)
Если школа долго не отвечает или отказывается выплачивать комиссию, рекрутер может привлечь администратора.
1. Рекрутер нажимает кнопку `Add Admin` в шапке чата.
2. Система находит доступного пользователя с `role = 'admin'` и записывает его ID в поле `admin_id` таблицы `chats`.
3. В чат отправляется системное сообщение.
4. Чат появляется в Inbox у администратора.
5. Админ может читать историю и писать сообщения.
6. После решения проблемы админ нажимает `Leave Chat`, поле `admin_id` очищается (NULL).

## 6. UI/UX Дизайн чата
- **Лаконичность:** Максимально чистый интерфейс. Скругленные углы (rounded-2xl).
- **Пузыри сообщений (Bubbles):** 
  - Мои сообщения: Цвет фона зависит от роли (Indigo для админа, Blue для рекрутера, Emerald для школы), текст белый. Выравнивание по правому краю.
  - Чужие сообщения: Фон серый (`bg-gray-100`), текст темный. Выравнивание по левому краю.
- **Контекст:** В шапке чата всегда отображается имя студента и ID заявки (кликабельно, ведет на детали заявки).
- **Защита от спама:** Кнопка отправки блокируется (disabled) во время выполнения запроса к БД (`isSending`).
